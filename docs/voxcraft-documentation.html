Introduction & Project Overview

1.1 Motivation & Vision

VoxCraft is an enterprise-grade, fully-local text-to-speech (TTS) studio engineered to bridge the gap between professional voiceover production and the accessibility of open-source tooling. The project was conceived as a response to a clear market deficiency: existing TTS solutions force users into one of two undesirable positions — either paying recurring subscription fees for cloud-based services that exfiltrate their content to remote servers, or settling for low-quality, under-featured open-source alternatives that lack the polish required for production content creation.

The core vision is radical in its simplicity: studio-quality voice synthesis that runs entirely on the user's own hardware, with zero API calls, zero telemetry, and zero recurring costs. VoxCraft achieves this by leveraging Supertonic 3, a 99-million-parameter ONNX model that performs inference directly on-device — no GPU required. This architectural choice means that a base-model M1 MacBook Air can generate 44.1 kHz 16-bit WAV audio suitable for Instagram Reels, YouTube videos, podcasts, audiobooks, and professional voiceover work, all without an internet connection after the initial model download.

The name “VoxCraft” reflects the dual nature of the application: vox (Latin for voice) and craft (the artisan's deliberate, skilled practice). Like a craftsman's workshop, the studio provides precise control over every parameter of voice generation — from voice character selection and language choice to speed modulation and denoising quality — while maintaining the immediacy and responsiveness of a local application.

Key Insight: VoxCraft is not merely a wrapper around Supertonic 3. It is a complete studio environment that adds voice management, expression tagging, batch processing, waveform visualization, generation history with full-text search, and a polished single-page application frontend — all orchestrated through a RESTful API that any application can consume.

1.3 Design Philosophy

VoxCraft's architecture is governed by four immutable design principles that inform every technical decision:

1.3.1 100% Local by Default

This is the project's defining constraint and its greatest strength. No synthesis data ever leaves the user's machine. The ONNX model runs inference directly on CPU via optimized C++ runtime bindings. There are no cloud API dependencies, no telemetry endpoints, no analytics beacons, and no license activation servers. The implication for the architecture is profound: every component must function in a fully air-gapped environment after initial model acquisition. This constraint drove the decision to use a file-based history system instead of a database, vanilla JavaScript instead of a framework with a build step, and FastAPI's built-in static file serving instead of a separate web server.

1.3.2 Zero Build Step

The frontend is a single HTML file with embedded CSS and JavaScript. There is no npm, no webpack, no Vite, no TypeScript compilation, and no minification pipeline. The entire studio application — all 741 lines of it — is served as a static file by the FastAPI backend. This choice was deliberate: it eliminates the entire category of frontend build tooling issues (version conflicts, broken CI pipelines, dependency hell), reduces the contribution barrier to near-zero (anyone can edit a single file), and ensures that the application loads faster than any framework-based SPA could ever achieve. WaveSurfer.js is loaded from a CDN via ES module import, but this is optional — it could be vendored for fully offline use.

1.3.3 Python-First Backend

The backend is pure Python 3.14 with type hints throughout. The codebase uses modern Python features (from __future__ import annotations for forward-reference resolution, NumPy for audio array manipulation, Pydantic v2 for request/response validation) while remaining approachable to Python developers of all skill levels. FastAPI was selected over Flask or Django for its automatic OpenAPI documentation generation, built-in async support, Pydantic integration, and performance characteristics (on par with Node.js frameworks due to Starlette/uvicorn underpinnings).

1.3.4 Fast by Default

Every interaction in the studio — from page load to voice generation to history search — is designed to feel instantaneous. The Supertonic engine is initialized eagerly on server startup (not lazily on first request). Voice styles are pre-cached in memory. The frontend makes parallel API calls on initialization to populate voices, languages, and health status simultaneously. The audio player uses the Web Audio API for sub-millisecond playback latency. There are no spinners for operations under 100ms; longer operations (synthesis) show progress feedback.

System Architecture

2.1 Architectural Overview

VoxCraft follows a monolithic single-process architecture with a clear separation of concerns between layers. This is not a microservices system, nor does it use a database — every component runs in a single Python process managed by Uvicorn. This architectural simplicity is a feature, not a limitation: it eliminates network latency between components, removes the need for inter-process communication, simplifies deployment to a single command, and makes the system trivially debuggable.

The system can be understood as a stack of five logical layers, each building on the one below it:

Figure 2.1 — VoxCraft Layered Architecture
Presentation Layer
Browser-based SPA (HTML/CSS/JS) · WaveSurfer.js · Fetch API · Web Audio API
↕ HTTP (REST)
Application Layer
FastAPI (app.py) · 10 REST endpoints · Pydantic v2 validation · CORS middleware
↕ Python API
Engine Layer
TTSEngine (tts_engine.py) · Voice cache · Audio export pipeline · History manager
↕ Python API
Inference Layer
Supertonic 3 (supertonic.TTS) · ONNX Runtime · 99M params · CPU-optimized
↕ Filesystem I/O
Persistence Layer
SQLite-backed storage · data/exports/ (WAV) · data/voxcraft.db (SQLite with WAL mode)

The key architectural insight is that the Engine Layer is the system's single source of truth for all TTS state. It owns the Supertonic client instance, maintains the voice style cache, performs synthesis, manages audio export with metadata, and tracks generation history. The Application Layer (FastAPI) is a thin orchestration facade that maps HTTP requests to Engine method calls and transforms results into JSON responses. This separation means the Engine can be tested independently of HTTP concerns, and the entire system could be re-fronted with a different protocol (gRPC, WebSocket, CLI) without touching the Engine.

2.3 Request-Response Lifecycle

The most performance-critical path in the system is the synthesis request. Understanding this lifecycle is essential for debugging, performance optimization, and contributing new features. The following trace follows a complete synthesis from browser to filesystem and back:

Figure 2.2 — Synthesis Request Lifecycle (POST /api/synthesize)
Phase 1: Request Ingestion (0–2 ms)
1. User types text, selects voice, clicks "Generate Speech" or presses Cmd+Enter
↓
2. Browser constructs JSON payload: {text, voice, language, speed, quality}
↓
3. fetch() sends POST to /api/synthesize with Content-Type: application/json
Phase 2: Validation (1–3 ms)
↓
4. FastAPI deserializes JSON → SynthesizeRequest Pydantic model; validates text length (1–5000 chars), voice name, language code, speed range (0.7–2.0), quality range (5–12)
↓
5. Validation failure → HTTP 400 with detail message; success → proceed to engine
Phase 3: Engine Readiness Check (0–1 ms)
↓
6. get_engine() returns singleton TTSEngine instance (or creates + initializes on first call)
↓
7. engine.is_ready check — if False (model not downloaded), return HTTP 503
Phase 4: Voice Resolution & Synthesis (100–5000 ms, dominant cost)
↓
8. engine._get_voice(voice_name) — hits in-memory cache or loads voice style from Supertonic
↓
9. engine.synthesize() → self._tts.synthesize(text, lang, voice_style, total_steps=quality, speed=speed)
↓
10. Supertonic runs ONNX inference — returns (wav: np.ndarray, duration: float)
Phase 5: Export & Persistence (5–20 ms)
↓
11. engine.export_for_platform() — generates UUID-based ID, timestamp, filename
↓
12. engine.save_audio() — writes WAV to data/exports/voxcraft_{id}.wav via SoundFile
13. data_store.save_export() — atomically writes metadata to SQLite (single source of truth)
Phase 6: Response (1–2 ms)
↓
15. SynthesizeResponse model constructed: {success, audio_url, duration_seconds, id, text, voice, language, sample_rate}
↓
16. FastAPI serializes to JSON; Uvicorn sends HTTP 200 response
↓
17. Browser receives response; loads audio URL into WaveSurfer; renders waveform; displays success toast

The total wall-clock time is dominated almost entirely by Phase 4 (ONNX inference), which typically ranges from 100 ms for short text at quality=5 to several seconds for maximum-length text at quality=12. All other phases combined contribute less than 30 ms to the total latency.

2.4 Data Flow & Persistence Model

VoxCraft uses a SQLite-backed persistence model with WAL mode for concurrent reads and atomic writes. Audio files are stored on disk; metadata lives in the database as the single source of truth.

data/exports/

Purpose: User-facing audio output

Contents: .wav (44.1kHz 16-bit mono)

Naming: voxcraft_{uuid12}_{timestamp}.wav

Cleanup: Manual via API; no automatic expiration

Served via: /api/audio/{filename}

data/voxcraft.db

Purpose: Metadata persistence & history

Contents: SQLite database (WAL mode, schema versioned)

Tables: exports (id, filename, text, voice, language, duration, ...), history (export_id, accessed_at)

Cleanup: DELETE via API removes DB row + audio file

Served via: /api/history

data/voices/

Purpose: Custom voice import target

Contents: .json (voice style definitions)

Naming: User-defined

Cleanup: Manual

Status: Directory created; feature planned (v1.2)

Design Note: v1.0.1 replaced the dual JSON directory approach with a SQLite-backed single source of truth. The data/history/ directory is deprecated — historical data is migrated to data/voxcraft.db on first run via migrate_from_json(). The exports table stores all metadata with indexes on created_at, voice, and language. The history table tracks access timestamps via a foreign key to exports. Deleting an export cascades to the history record and removes the WAV file atomically.

Backend API Reference

3.1 API Design Conventions

VoxCraft's REST API adheres to a consistent set of design conventions that make it predictable, self-documenting, and easy to integrate with external tools and scripts:

  • Base URL: All endpoints are prefixed with /api/ to namespace them separately from the frontend static file routes. The root path serves the SPA index.html.
  • Content Type: All request and response bodies use application/json. The API does not accept form-encoded or multipart payloads for synthesis (multipart is only used by the Swagger UI's built-in testing interface).
  • Success Envelope: Mutation responses include a success: true field. Read-only endpoints return data directly (list of voices, language dict, etc.) without an envelope.
  • Error Envelope: Errors return {"success": false, "error": "human-readable message"} for mutation endpoints. FastAPI's built-in validation errors return the standard {"detail": [...]} format with field-level error locations.
  • HTTP Semantics: GET for reads, POST for creations/actions, DELETE for deletions. The API does not use PUT or PATCH — there are no updatable resources in v1.0.
  • OpenAPI Documentation: Available at /docs (Swagger UI) and /redoc (ReDoc). These are auto-generated by FastAPI from Pydantic model schemas and docstrings.
  • CORS: All origins are permitted (allow_origins=["*"]) for local development convenience. For production deployments, this should be restricted.

3.2 GET /api/health — Health Check

GET /api/health

Returns the current status of the VoxCraft server and its TTS engine. This is the first endpoint the frontend calls on load and is used to display the online/offline indicator in the top bar. No authentication is required.

Response Schema (200 OK)

{
  "status": "healthy" | "initializing",
  "engine_ready": true | false,
  "voices_available": 10,
  "languages_available": 31
}
FieldTypeDescription
statusstring"healthy" if the engine has initialized successfully; "initializing" if the ONNX model is still downloading or the engine failed to initialize.
engine_readybooleanDirect reflection of TTSEngine.is_ready. When false, synthesis requests will return HTTP 503.
voices_availableintegerNumber of preset voice styles loaded (always 10).
languages_availableintegerNumber of supported languages (always 31).

Example

// Request
curl http://localhost:8765/api/health

// Response
{"status":"healthy","engine_ready":true,"voices_available":10,"languages_available":31}

Error Cases

This endpoint does not return error responses. If the server is unreachable, the client treats it as offline. If the engine is not ready, the response body indicates this with engine_ready: false rather than an error status code.

Text-to-Speech Engine

4.1 Supertonic 3 Integration

The TTS engine is the heart of VoxCraft — a carefully designed abstraction layer over Supertonic 3 that provides voice management, audio export, metadata tracking, and history persistence. It is implemented across two files: backend/tts_engine.py (315 lines, synthesis + voice management) and backend/data_store.py (234 lines, SQLite persistence). The engine exports a singleton accessor function (get_engine()) that all API endpoints use.

Supertonic 3 is a 99-million-parameter ONNX model that performs text-to-speech inference entirely on CPU. It supports 31 languages and custom voice styles defined via JSON configuration files. The model is downloaded once from Hugging Face (~/.cache/supertonic3/, approximately 400 MB) and loaded into memory on first use. The Python API (supertonic.TTS) provides a simple interface:

from supertonic import TTS

tts = TTS(auto_download=True)              # Initialize; download model if needed
voice = tts.get_voice_style(voice_name="M1")  # Load a voice style
wav, duration = tts.synthesize(               # Generate audio
    text="Hello world",
    lang="en",
    voice_style=voice,
    total_steps=8,   # Denoising steps (quality)
    speed=1.05,      # Speech rate
)
# wav: np.ndarray (mono audio samples)
# duration: np.ndarray with one element (float duration in seconds)
ONNX Runtime: Supertonic 3 uses ONNX Runtime under the hood for model inference. This means it benefits from ONNX Runtime's CPU optimizations (threading, vectorization) and can run on any platform that supports ONNX Runtime — macOS (x86_64 and ARM64), Windows, and Linux. No GPU or CUDA installation is required.

4.2 TTSEngine Class Architecture

The TTSEngine class encapsulates all Supertonic interaction behind a clean, testable interface. Its design follows the Facade pattern: it hides the complexity of model loading, voice caching, audio I/O, and metadata management behind a handful of high-level methods.

Constructor & Initialization

class TTSEngine:
    def __init__(self, auto_download: bool = True):
        self._tts: TTS | None = None
        self._voice_cache: dict[str, Any] = {}
        self._ready = False

        try:
            self._tts = TTS(auto_download=auto_download)
            self._ready = True
            self._get_voice("M1")  # Pre-cache default voice
        except Exception as e:
            print(f"TTS Engine init warning: {e}")

The constructor is eager, not lazy. It initializes the Supertonic client and pre-caches the default voice (M1) immediately. This means the first synthesis request after server startup does not pay the initialization penalty — the model is already loaded and the voice is already cached. If initialization fails (e.g., network error during model download), the engine enters a degraded state where is_ready returns False and all synthesis requests return HTTP 503. The server itself remains operational, serving health checks, voice lists, and the frontend.

4.3 Voice Style Caching

Voice styles in Supertonic are loaded from JSON configuration files. Loading a voice style is not free — it involves file I/O and parsing — so VoxCraft maintains an in-memory cache (_voice_cache) that maps voice names to their loaded style objects.

The caching strategy is lazy population with eager priming: M1 is loaded in the constructor to make the first synthesis request fast; all other voices are loaded on first use. Once loaded, a voice stays in the cache for the lifetime of the server process. With only 10 preset voices, the memory overhead is negligible (each voice style JSON is a few KB).

If custom voice import is implemented (v1.2 roadmap), the cache strategy should be revisited — a large number of custom voices could justify an LRU eviction policy.

4.4 Synthesis Pipeline — In Detail

The synthesize() method implements a six-stage pipeline:

  1. Readiness Gate: If self._ready is False or self._tts is None, raise RuntimeError. This is the only error that indicates a system-level problem rather than a user input problem.
  2. Input Validation: Check that text is non-empty and contains at least one non-whitespace character. Raise ValueError with a descriptive message if validation fails. Note that the FastAPI layer also validates text length (1–5000 via Pydantic), so this is a defense-in-depth check for direct engine usage.
  3. Voice Resolution: Call _get_voice(voice) — this is a cache-hit-or-load operation that returns the Supertonic voice style object for the requested voice name.
  4. Parameter Clamping: quality is clamped to [5, 12] and speed to [0.7, 2.0]. This is defense-in-depth; the FastAPI layer already enforces these ranges via Pydantic Field(ge=..., le=...) constraints.
  5. Model Inference: Delegates to self._tts.synthesize(text, lang, voice_style, total_steps=quality, speed=speed). This is the blocking, CPU-bound operation that dominates wall-clock time. Supertonic runs a diffusion-based denoising process: each denoising step refines the audio, and the total_steps parameter controls the number of refinement iterations.
  6. Return Value Extraction: Supertonic returns a tuple of (wav: np.ndarray, duration: np.ndarray). The duration is a single-element array; VoxCraft extracts the scalar value as float(duration[0]) for cleaner API responses.

4.5 Audio Export & Metadata System

The export_for_platform() method is the bridge between raw synthesis output and user-visible artifacts. It performs five operations:

  1. ID Generation: Creates a 12-character unique ID from the hex representation of a UUID4 (uuid.uuid4().hex[:12]). Provides 48 bits of entropy — collision probability under 0.001% at one million generations. Short enough for human-readable filenames, long enough for statistical uniqueness.
  2. Timestamp Generation: Creates a UTC timestamp in YYYYMMDD_HHMMSS format. The UTC timezone ensures consistent ordering regardless of the user's local timezone.
  3. Filename Construction: Builds voxcraft_{id}_{timestamp}.{format}. Example: voxcraft_a1b2c3d4e5f6_20260614_120000.wav.
  4. Audio Persistence: Calls save_audio() which:
    • Creates parent directories if needed
    • Squeezes multi-dimensional arrays to mono
    • Writes the WAV file via soundfile.write() at 44.1kHz
    • Returns the absolute path to the written file
  5. Metadata Persistence: Calls data_store.save_export() which atomically writes the metadata record to the SQLite exports table with an associated history access log entry. A single database transaction ensures both are written together. No duplicate JSON files.

4.6 Engine Singleton Pattern

The engine uses a module-level singleton pattern:

# Global engine singleton
_engine: TTSEngine | None = None

def get_engine() -> TTSEngine:
    global _engine
    if _engine is None:
        _engine = TTSEngine()
    return _engine

This ensures that: (1) exactly one Supertonic client exists in the process (avoiding duplicate model loads), (2) the voice cache is shared across all API requests, (3) initialization happens on first access (which is at import time for get_engine if called eagerly, or on the first API request if accessed lazily), and (4) the engine is trivially testable by resetting the _engine global in test fixtures.

Expression Tag System

7.1 Tag Semantics

Expression tags are inline markup tokens that instruct the TTS engine to inject non-verbal vocal expressions into the synthesized speech. They are inspired by SSML (Speech Synthesis Markup Language) but use a simplified syntax compatible with raw text editing — no XML namespace declarations, no closing tags required for some operations.

VoxCraft supports six expression tags in v1.0:

TagMarkupSemantic EffectUse Case
Laugh<laugh>Inserts a natural laugh sound at the tag position. The laugh is generated in the same voice as the surrounding speech, maintaining character consistency.Podcasts, casual content, reaction videos
Breath<breath>Inserts an audible breath. Useful for creating natural speech rhythms, pauses, and emotional emphasis.Audiobooks, dramatic narration, meditation guides
Sigh<sigh>Inserts a sigh expression. Conveys emotion (relief, frustration, weariness) without explicit dialogue.Storytelling, character dialogue, emotional content
Whisper<whisper>Wraps text in whispered delivery. Can be used inline — <whisper>this is secret</whisper> — or as a standalone toggle.ASMR content, dramatic reveals, intimate narration

7.2 Tag Processing Pipeline

Expression tags are passed through to Supertonic 3 without preprocessing by VoxCraft. The tag parsing and rendering are handled entirely by the TTS model during inference. This means:

  • No tag validation: VoxCraft does not check for well-formed tags (matching open/close pairs). Invalid or misspelled tags are passed to Supertonic and either ignored or rendered as literal text, depending on Supertonic's internal behavior.
  • No tag escaping: There is currently no mechanism to render the literal text <laugh> in speech output. A future SSML implementation (v1.2 roadmap) would address this with proper escaping.
  • Position-independent: Tags can be placed anywhere in the text — beginning, middle, or end. The engine processes them in document order.

7.3 Tag Composition & Nesting

Tags can be composed freely. For example:

<breath>I can't believe this is happening. <sigh>What a day. <laugh>

This would produce a breath sound, followed by the first sentence, then a sigh, then "What a day," then a laugh. The frontend tag toolbar makes it easy to insert tags at the cursor position: clicking a tag button inserts the tag, or wraps the currently selected text with opening and closing tags (for whisper mode).

Pro Tip: For whispered delivery, select the text you want whispered and click the whisper tag button. The frontend automatically wraps it: <whisper>selected text</whisper>.

Frontend Architecture

8.1 Single-Page Application Design

The VoxCraft frontend is a pure vanilla JavaScript single-page application (SPA) implemented in a single HTML file (frontend/index.html, 755 lines). It uses no frameworks, no build tools, no package manager, and no transpilation step. The entire application — HTML structure, CSS styling, and JavaScript logic — is contained in one file that the FastAPI backend serves as a static asset. Supporting assets (Inter font in 5 weights, WaveSurfer.js v7) are vendored in frontend/fonts/ and frontend/assets/ — no CDN dependencies, fully offline-capable after initial clone.

This architectural choice is unusual in modern web development but is a deliberate design decision rooted in VoxCraft's core philosophy:

  • Zero build step: No npm install, no webpack config, no TypeScript compilation. Edit the file, refresh the browser. This dramatically lowers the contribution barrier and eliminates entire categories of frontend build issues.
  • Minimal dependencies: Only one external dependency — WaveSurfer.js v7, vendored in the repository as an ES module. Everything else (state management, routing, DOM manipulation) uses native browser APIs.
  • Instant load: A single 31 KB HTML file loads faster than any framework-based SPA can bootstrap. No JavaScript bundles to parse, no virtual DOM to initialize.
  • Future-proof: Vanilla JS ages gracefully. Framework APIs change; native browser APIs are stable and backward-compatible for decades.
Contribution Constraint: Replacing the vanilla JS frontend with React, Vue, Svelte, or any other framework is explicitly out of scope. This is a project-level design constraint documented in CONTRIBUTING.md. The zero-build-step philosophy is non-negotiable for v1.x and v2.0.

The Web Audio backend provides sub-millisecond playback latency and allows precise seeking. Waveform normalization ensures consistent visual representation regardless of audio loudness. The minPxPerSec: 30 setting guarantees that short audio clips still produce a detailed waveform.

Data Persistence & Storage

9.1 Directory Structure

VoxCraft creates and manages a data/ directory at the project root (or at a custom path set via the VOXCRAFT_DATA_DIR environment variable). The directory structure is created automatically on engine initialization:

voxcraft/
└── data/
    ├── exports/    ← Generated WAV files + metadata JSON
    ├── history/    ← Generation history records (JSON)
    └── voices/     ← Custom voice import target (future use)

Each subdirectory is created with mkdir(parents=True, exist_ok=True), ensuring that the engine can start even if the directories don't exist yet. This also means that deleting the data/ directory and restarting the server is a valid (albeit destructive) reset operation.

9.2 Export Format Specification

Generated audio files follow a strict naming convention:

voxcraft_{generation_id}_{timestamp}.wav

Where:

  • generation_id = first 8 characters of a UUID4 (hex, lowercase)
  • timestamp = UTC timestamp in YYYYMMDD_HHMMSS format

Audio files are written in WAV format with the following properties:

PropertyValueRationale
Sample Rate44,100 HzCD-quality audio standard; compatible with all platforms
Bit Depth16-bitStandard for distribution; balances quality and file size
Channels1 (Mono)Speech is naturally monophonic; stereo would double file size with no benefit
CodecPCM (uncompressed)Lossless; universally supported; suitable for further processing
File Extension.wavStandard WAV container; recognized by all audio software

9.3 Metadata Schema

Each generation produces a JSON metadata file with the same base filename as the audio:

// data/exports/voxcraft_a1b2c3d4_20260614_120000.json
// data/history/voxcraft_a1b2c3d4_20260614_120000.json

{
  "id": "a1b2c3d4",                         // 8-char generation UUID
  "filename": "voxcraft_a1b2c3d4_20260614_120000",  // Base filename (no extension)
  "text": "The original input text",          // Full text that was synthesized
  "voice": "M1",                              // Voice style used
  "language": "en",                           // Language code used
  "duration_seconds": 3.45,                   // Audio duration (rounded to 2 decimal places)
  "format": "wav",                            // Output format
  "created_at": "20260614_120000",            // UTC timestamp
  "audio_path": "/absolute/path/to/file.wav"  // Full filesystem path to WAV
}
JSON Schema Validation: The metadata files are not validated against a formal JSON Schema at write time. They are assumed to be correct by construction (the export_for_platform() method generates them). However, the get_history() method wraps JSON parsing in try/except to gracefully skip corrupted history files rather than crashing the request.

9.4 History Management

Generation history is maintained as individual JSON files in data/history/. The get_history() method reads these files on demand:

  1. Glob all .json files in the history directory
  2. Sort by filesystem modification time (descending)
  3. Parse the first N files (capped by limit parameter)
  4. Skip any file that fails JSON parsing (corrupted or non-metadata JSON)
  5. Return the list of parsed metadata dicts

There is no in-memory history cache — every request performs filesystem I/O. For the current scale (up to 200 records), this is imperceptible (~5–10 ms for a full directory scan). If history grows larger, an in-memory ring buffer or embedded SQLite database should replace this approach.

By default, VoxCraft binds to 0.0.0.0:8765, making it accessible from any device on the local network. For security-conscious deployments, setting VOXCRAFT_HOST=127.0.0.1 restricts access to localhost only.

The port 8765 was selected because it falls in the user-range (1024–49151), does not conflict with common development server ports, and is easy to remember (ascending digits: 8-7-6-5).

Security Architecture

15.1 Security Design Principles

VoxCraft's security posture is built on three principles:

  1. No Data Exfiltration: The application makes zero outbound network calls for synthesis. After the one-time model download, VoxCraft can operate in a fully air-gapped environment. There is no telemetry, no analytics, no crash reporting, no license activation. All frontend assets (fonts, waveform renderer) are vendored in the repository — the browser makes zero external requests after page load.
  2. Minimal Attack Surface: VoxCraft exposes only a single HTTP server on a configurable port. There is no authentication system to attack, no database to inject, no user accounts to enumerate, no password reset flow to exploit. The API is designed for localhost use and does not implement authentication in v1.0 (API key authentication is planned for v2.0).
  3. Defense in Depth: Path traversal protections are implemented at multiple layers. The CORS configuration is permissive by default but documented for hardening in production deployments. Input validation happens at both the Pydantic layer and the engine layer.

15.3 Path Traversal Mitigation

The audio serving endpoint is the most sensitive path in the system — it reads files from disk based on user-supplied filenames. The implementation uses os.path.basename() as the primary defense:

safe_name = os.path.basename(unquote(filename))
audio_path = EXPORTS_DIR / safe_name

This strips any directory traversal payload (e.g., ../../../etc/passwd becomes just passwd). Combined with the requirement that the file must exist in EXPORTS_DIR, this effectively eliminates path traversal as an attack vector.

The frontend static file serving endpoint uses a different approach — Path.relative_to() — which verifies that the resolved path is a descendant of the intended directory:

requested = FRONTEND_DIR / full_path
try:
    requested.relative_to(FRONTEND_DIR)
except ValueError:
    raise HTTPException(status_code=404)

15.4 CORS & Network Security

The current CORS configuration permits all origins:

app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

This is appropriate for a localhost development tool but should be restricted for any deployment where the server is accessible from the network. For production deployments, the recommended configuration is:

allow_origins=["https://your-domain.com"]

Critical vulnerabilities will be patched as quickly as possible, and disclosure will be coordinated with the reporter.

Appendix G: Glossary of Terms

TermDefinition
ASGIAsynchronous Server Gateway Interface — the modern Python standard for async web servers. Uvicorn is an ASGI server implementation.
Denoising StepsIterations of the diffusion-based audio refinement process. More steps produce cleaner audio at the cost of longer generation time.
Expression TagInline markup token that instructs the TTS engine to produce a non-verbal vocal expression (laugh, breath, etc.).
FastAPIModern Python web framework built on Starlette and Pydantic. Provides automatic OpenAPI documentation generation.
ISO 639-1International standard for two-letter language codes. VoxCraft uses these codes for language identification.
ONNXOpen Neural Network Exchange — an open format for machine learning models that enables cross-platform inference.
PCMPulse-Code Modulation — uncompressed digital audio representation. WAV files use PCM encoding.
PydanticPython data validation library using type annotations. VoxCraft uses Pydantic v2 for request/response schemas.
RTLRight-to-Left — writing direction used by Arabic, Hebrew, and other Semitic scripts.
SPASingle-Page Application — a web application that loads a single HTML page and dynamically updates content without full page reloads.
SSMLSpeech Synthesis Markup Language — XML-based markup for controlling TTS output. Planned for VoxCraft v1.2.

UvicornLightning-fast ASGI server implementation. Runs the FastAPI application and serves both API and static files.

Web Audio APIBrowser API for processing and playing audio with low latency. Used by WaveSurfer.js for waveform rendering.
